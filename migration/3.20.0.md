# Migrate from 3.18.1 to 3.20.0

This release tracks an upstream proto update. Two `bytebase_setting` (`settings/WORKSPACE_PROFILE`) attributes change shape — one is a **breaking rename**, one **replaces a field with a nested block** — and `bytebase_instance` gains an optional `token_type` on the Vault external-secret block.

## Breaking change: `maximum_role_expiration_in_seconds` renamed

Inside `bytebase_setting`'s `workspace_profile` block, `maximum_role_expiration_in_seconds` is renamed to `maximum_request_expiration_in_seconds`. The underlying proto field was renamed (`MaximumRoleExpiration` → `MaximumRequestExpiration`, same field number and `Duration` type), and the attribute follows. The semantics are unchanged: the max expiration duration applies to role grants and data access requests, and a value `<= 0` removes the setting (no limit).

Update your config:

```hcl
resource "bytebase_setting" "workspace_profile" {
  name = "settings/WORKSPACE_PROFILE"

  workspace_profile {
-   maximum_role_expiration_in_seconds    = 2592000 # 30 days
+   maximum_request_expiration_in_seconds = 2592000 # 30 days
  }
}
```

Configs that still reference the old attribute name fail at plan time with an "Unsupported argument" error.

## Breaking change: announcement `level` replaced by a `theme` block

The `announcement` block inside `workspace_profile` no longer has a `level` (String) attribute. Upstream replaced the alert-level enum with banner colors, so the provider now exposes a `theme` block:

```hcl
resource "bytebase_setting" "workspace_profile" {
  name = "settings/WORKSPACE_PROFILE"

  workspace_profile {
    announcement {
      text = "Scheduled maintenance this weekend."
      link = "https://bytebase.example.com/maintenance"

-     level = "WARNING"
+     # Banner colors in "r g b" format. The info/warning/critical presets are a
+     # frontend-only concept that simply seeds these colors.
+     theme {
+       background = "255 247 224"
+       text       = "151 90 22"
+     }
    }
  }
}
```

`theme` is an optional singleton block with two optional string attributes, `background` and `text`, both in `"r g b"` format. The built-in `info` / `warning` / `critical` presets are a frontend-only concept that seed these colors; the store holds only the raw colors. Remove any `level = "…"` line — it's now an unsupported argument.

## Feature: Vault external-secret `token_type`

Each connection in a `bytebase_instance`'s `data_sources` block can back its credential with an `external_secret`. For the HashiCorp Vault source, the `vault` block gains an optional `token_type` describing how the Bytebase server should interpret the `token` value:

| Value | Meaning |
| --- | --- |
| `PLAIN` | The literal token value (default). |
| `ENVIRONMENT` | The name of an environment variable on the Bytebase server holding the token. |
| `FILE` | A path to a file on the Bytebase server holding the token. |

```hcl
data_sources {
  external_secret {
    vault {
      token      = var.vault_token
      token_type = "PLAIN"
    }
  }
}
```

The field is optional; omitting it preserves prior behavior (treated as `PLAIN`). Unlike `token`, which is write-only/sensitive, `token_type` is returned on GET, so it surfaces as a read-only `token_type` (String) in the `bytebase_instance` data source.

## No other changes

No Terraform CLI version bump beyond the 1.11 minimum already required since 3.18.0. Resources and data sources not listed above are unaffected.
