# Migrate from 3.18.0 to 3.18.1

No required configuration changes for most users. The change in this release affects only `bytebase_project.webhooks`, which no longer stores plaintext `url` values in Terraform state and gains a few stricter rules.

## Behavior changes

### `webhooks[].url` is now stored as a digest

The `url` attribute inside `bytebase_project.webhooks` remains configured inline, but the provider now stores only a SHA-256 digest of the URL in Terraform state. The plaintext URL is read from raw config during create/update and sent to the Bytebase API, but it is not persisted in state.

The minimum Terraform CLI requirement (1.11) was already established in 3.18.0; no further version pin needed.

After upgrading, the **first** plan may show a one-time refresh diff replacing previously-stored plaintext webhook URLs with their digests. No API call is made for the diff itself; the secrets on the server are unchanged.

### `webhooks` is now a list, not a set

The schema type flips from `TypeSet` to `TypeList`. HCL syntax for users is unchanged — `webhooks { … }` blocks and `webhooks = [ … ]` attribute assignment both continue to work because `ConfigMode = SchemaConfigModeAttr` is preserved.

What does change: **order matters now**. If your config declares webhooks in a different order than the server returned them at last apply, Terraform will show a diff on the affected indices. On the first apply after upgrade, you may see one no-op churn-diff while the order settles, or you may need to reorder your config to match what's already on the server.

This change is necessary because the previous `TypeSet` identity included the plaintext URL among other fields. Once the plaintext URL is no longer stored in state, set-hash identity becomes ambiguous.

### Duplicate `(title, type)` webhooks are rejected at plan time

Because identity for updates now uses the `(title, type)` pair (plaintext URL is no longer available in state), two webhooks with the same title and type within a single project would be ambiguous to match. The provider rejects this at plan time:

```
Error: duplicate webhook with title="release alerts" and type="SLACK";
(title, type) must be unique within a project because plaintext url is not
stored in state and (title, type) is the identity used for matching
```

Rename one of the colliding webhooks before applying.

## Rotation flow

To rotate a webhook URL (e.g., regenerate the secret on the Slack side):

1. Update the URL value in your config (variable / `tfvars` / Vault / TFC variable).
2. Run `terraform apply`.
3. Terraform detects that the URL digest changed, and the provider reads the new plaintext URL from raw config and sends it to Bytebase.

URL-only rotations are detected because Terraform compares URL digests, not plaintext URLs.

## Recommended: store URLs outside Terraform

Because the plaintext URL must be present in your config on every apply, move webhook URLs to a managed source: input variables, `tfvars`, Vault, Terraform Cloud variables, or your existing secrets pipeline. Storing them inline in `*.tf` files committed to git defeats the security benefit.

## Forward-compatibility note

State written by 3.18.1 contains a SHA-256 digest in `webhooks[].url`, not the plaintext URL. A 3.17.x or older provider reading 3.18.1 state will compare that digest as if it were a URL and may show an unexpected diff. Pin one direction during upgrade/downgrade.

## New: `bytebase_setting` supports `APP_IM`

`bytebase_setting` accepts `name = "settings/APP_IM"` with a new `app_im` block that configures workspace IM integrations: Slack, Feishu, WeCom, Lark, DingTalk, and Microsoft Teams.

```hcl
resource "bytebase_setting" "app_im" {
  name = "settings/APP_IM"

  app_im {
    slack {
      token = var.slack_token
    }
    feishu {
      app_id     = var.feishu_app_id
      app_secret = var.feishu_app_secret
    }
  }
}
```

Each IM block is an optional singleton. Including a block enables that integration; omitting it disables it on the next apply. Server-side rejection of duplicate IM types is enforced naturally by HCL — you can't write two `slack` blocks in the same `app_im` parent.

**Every field — secrets AND non-secret identifiers — is write-only.** The Bytebase server strips all payload fields on GET (`convertToAppIMSetting` in `backend/api/v1/setting_service_converter.go` constructs empty payload structs for every IM type), so non-write-only fields would produce perpetual drift. Practical implications, same as the other 3.18.0/3.18.1 write-only fields:

- All field values must be present in config on every apply (use input variables / tfvars / Vault / TFC variables).
- State persists only the *set of configured IM types*. Field values are always absent.
- `terraform plan` cannot detect server-side drift on any field — drift on the *set of configured types* IS detected.
- Rotating a credential: change the value in config and apply. If the rotation is for a single field and Terraform shows no diff, also touch another tracked attribute on the same IM block (e.g. toggle a different IM type or use `-replace`).

`data.bytebase_setting` also supports `name = "settings/APP_IM"`. Because the server returns empty payloads, the data source surfaces only block presence — useful for assertions like "is Slack configured on this workspace?" Field values in the data source are always `""`.
