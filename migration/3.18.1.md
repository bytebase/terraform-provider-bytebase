# Migrate from 3.18.0 to 3.18.1

No required configuration changes for most users. The change in this release affects only `bytebase_project.webhooks`, which gains a write-only `url` and a few stricter rules.

## Behavior changes

### `webhooks[].url` is now write-only

The `url` attribute inside `bytebase_project.webhooks` converts from `Sensitive: true` to `WriteOnly: true`. This is the same conversion 3.18.0 applied to `bytebase_user.password`, `bytebase_workspace.license`, and the three IDP secrets ŌĆö see `migration/3.18.0.md` for the general behavior model. Same caveats apply: the value is read from raw config on every apply, never written to state, and server-side drift on the URL is no longer detected.

The minimum Terraform CLI requirement (1.11) was already established in 3.18.0; no further version pin needed.

After upgrading, the **first** plan will show a one-time refresh diff zeroing out the previously-stored webhook URLs from state. No API call is made for the diff itself; the secrets on the server are unchanged.

### `webhooks` is now a list, not a set

The schema type flips from `TypeSet` to `TypeList`. HCL syntax for users is unchanged ŌĆö `webhooks { ŌĆ” }` blocks and `webhooks = [ ŌĆ” ]` attribute assignment both continue to work because `ConfigMode = SchemaConfigModeAttr` is preserved.

What does change: **order matters now**. If your config declares webhooks in a different order than the server returned them at last apply, Terraform will show a diff on the affected indices. On the first apply after upgrade, you may see one no-op churn-diff while the order settles, or you may need to reorder your config to match what's already on the server.

This change is necessary because the previous `TypeSet` identity was hashed from URL among other fields; once URL is absent from state, set-hash identity becomes ambiguous.

### Duplicate `(title, type)` webhooks are rejected at plan time

Because identity for diffing now uses the `(title, type)` pair (URL is no longer available in state), two webhooks with the same title and type within a single project would be ambiguous to match. The provider rejects this at plan time:

```
Error: duplicate webhook with title="release alerts" and type="SLACK";
(title, type) must be unique within a project because url is write-only and
(title, type) is the identity used for diffing
```

Rename one of the colliding webhooks before applying.

## Rotation flow

To rotate a webhook URL (e.g., regenerate the secret on the Slack side):

1. Update the URL value in your config (variable / `tfvars` / Vault / TFC variable).
2. Run `terraform apply`.
3. The provider reads the new URL from raw config and sends it to Bytebase.

If the only change is the URL, Terraform may suppress the diff display (URL is write-only). If your apply seems to be a no-op when you intended to rotate, also adjust another tracked field on the webhook (e.g. add/remove a `notification_types` entry, or change `direct_message`) to force the resource through an update ŌĆö same workaround as the existing 3.18.0 write-only fields.

## Recommended: store URLs outside Terraform

Because the URL must be present in your config on every apply (no state copy exists), move webhook URLs to a managed source: input variables, `tfvars`, Vault, Terraform Cloud variables, or your existing secrets pipeline. Storing them inline in `*.tf` files committed to git defeats the security benefit.

## Forward-compatibility note

State written by 3.18.1 contains no `url` key inside webhook elements. A 3.17.x or older provider reading 3.18.1 state will treat the URL as missing/required and fail to plan. As with the 3.18.0 write-only conversion, pin one direction.

## New: `bytebase_setting` supports `APP_IM`

`bytebase_setting` accepts `name = "settings/APP_IM"` with a new `app_im` block that configures workspace IM integrations: Slack, Feishu, WeCom, Lark, DingTalk, and Microsoft Teams.

```hcl
resource "bytebase_setting" "app_im" {
  name = "settings/APP_IM"

  app_im {
    slack {
      token = var.slack_token
    }
    feishu {
      app_id     = var.feishu_app_id
      app_secret = var.feishu_app_secret
    }
  }
}
```

Each IM block is an optional singleton. Including a block enables that integration; omitting it disables it on the next apply. Server-side rejection of duplicate IM types is enforced naturally by HCL ŌĆö you can't write two `slack` blocks in the same `app_im` parent.

**Every field ŌĆö secrets AND non-secret identifiers ŌĆö is write-only.** The Bytebase server strips all payload fields on GET (`convertToAppIMSetting` in `backend/api/v1/setting_service_converter.go` constructs empty payload structs for every IM type), so non-write-only fields would produce perpetual drift. Practical implications, same as the other 3.18.0/3.18.1 write-only fields:

- All field values must be present in config on every apply (use input variables / tfvars / Vault / TFC variables).
- State persists only the *set of configured IM types*. Field values are always absent.
- `terraform plan` cannot detect server-side drift on any field ŌĆö drift on the *set of configured types* IS detected.
- Rotating a credential: change the value in config and apply. If the rotation is for a single field and Terraform shows no diff, also touch another tracked attribute on the same IM block (e.g. toggle a different IM type or use `-replace`).

`data.bytebase_setting` also supports `name = "settings/APP_IM"`. Because the server returns empty payloads, the data source surfaces only block presence ŌĆö useful for assertions like "is Slack configured on this workspace?" Field values in the data source are always `""`.
