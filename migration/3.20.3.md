# Migrate from 3.20.2 to 3.20.3

No required configuration changes for most users. This release fills several upstream proto field gaps in the provider schema so Terraform can configure or read fields that Bytebase already exposes.

After upgrading, the first refresh or plan may show newly-populated computed fields in state. Those are state-only additions unless you also change configuration.

## `bytebase_instance` labels

`bytebase_instance` now supports the upstream `Instance.labels` field:

```hcl
resource "bytebase_instance" "mysql" {
  resource_id = "mysql-prod"
  environment = "environments/prod"
  title       = "MySQL Prod"
  engine      = "MYSQL"
  activation  = true

  labels = {
    region = "us-central1"
    team   = "platform"
  }

  data_sources {
    id       = "admin"
    type     = "ADMIN"
    host     = "mysql.example.com"
    port     = "3306"
    username = "admin"
    password = var.mysql_password
  }
}
```

The field is optional and computed on the resource. `data.bytebase_instance` and `data.bytebase_instance_list` also expose `labels` as read-only data.

If an existing Bytebase instance already has labels that Terraform did not previously track, the next refresh will populate them into Terraform state. To manage labels from Terraform, add the `labels` map to the resource config; to leave server-managed labels untouched, omit the field.

## Cloud SQL IP type

`bytebase_instance.data_sources` now supports `cloud_sql_ip_type`, matching the upstream `DataSource.cloud_sql_ip_type` field used by Google Cloud SQL connections:

```hcl
resource "bytebase_instance" "cloudsql" {
  resource_id = "cloudsql-prod"
  environment = "environments/prod"
  title       = "Cloud SQL Prod"
  engine      = "POSTGRES"
  activation  = true

  data_sources {
    id                  = "admin"
    type                = "ADMIN"
    authentication_type = "GOOGLE_CLOUD_SQL_IAM"
    cloud_sql_ip_type   = "PRIVATE"
  }
}
```

Supported values are `PUBLIC`, `PRIVATE`, and `PSC`. The field is optional; omitting it preserves previous behavior. Instance data sources and instance-list data sources expose the returned value as read-only data.

## Additional read-only instance fields

`bytebase_instance`, `data.bytebase_instance`, and `data.bytebase_instance_list.instances` now expose these read-only fields from the upstream `Instance` protocol message:

- `state`
- `last_sync_time`
- `roles`

These fields are computed by Bytebase. They are available for outputs, assertions, and downstream Terraform expressions, but they are not configurable.

## Additional read-only project fields

`data.bytebase_project` and `data.bytebase_project_list.projects` now expose read-only parity for fields that the `bytebase_project` resource already supports:

- `execution_retry_policy`
- `ci_sampling_size`
- `parallel_tasks_per_rollout`

No resource configuration changes are required.

## Additional read-only database fields

`bytebase_database`, `data.bytebase_database`, and `data.bytebase_database_list.databases` now expose these read-only fields from the upstream `Database` protocol message:

- `release`
- `effective_environment`
- `instance_resource`
- `backup_available`
- `sync_status`
- `sync_error`

`instance_resource` includes the database's backing instance metadata and its data sources as returned by Bytebase. These fields are computed and should be used for inspection or outputs only.

## No state migration required

This release does not require `terraform state mv`, `terraform state rm`, or re-importing resources. New fields are either optional configuration fields or computed read fields populated during refresh.
