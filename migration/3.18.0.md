# Migrate from 3.17.4 to 3.18.0

No schema-breaking changes. Existing configurations apply cleanly, but five sensitive fields now use Terraform's write-only mechanism (see below), which raises the minimum Terraform CLI version.

## Terraform CLI requirement

Minimum **Terraform CLI 1.11** (was 1.3.5). Earlier CLIs do not understand the `WriteOnly` schema attribute and will reject the provider during `terraform init`. Pin in your root module:

```hcl
terraform {
  required_version = ">= 1.11"
}
```

## Behavior changes

### Five sensitive fields are now write-only

The following secret-bearing fields converted from `Sensitive: true` to `WriteOnly: true`:

| Resource | Field |
| --- | --- |
| `bytebase_user` | `password` |
| `bytebase_workspace` | `license` |
| `bytebase_idp` | `oauth2_config.client_secret` |
| `bytebase_idp` | `oidc_config.client_secret` |
| `bytebase_idp` | `ldap_config.bind_password` |

**What `Sensitive` did before:** value was redacted in plan output but still persisted in `terraform.tfstate`.

**What `WriteOnly` does now:** value is read from the config on each apply, sent to the Bytebase API, and never written to state. The state column for these attributes is always `null`.

Practical implications:

- The value must be present in your config (or an input variable / `tfvars` / external secret source) on every apply. There is no `terraform plan -refresh-only` path that re-reads them — Bytebase doesn't return secrets, and the provider has no state copy to compare against.
- After upgrading, the **first** plan will show a one-time refresh diff zeroing out the previously-stored values from state. No API call is made for the diff itself; the secret on the server is unchanged.
- `terraform plan` will no longer detect drift on these fields — the diff is suppressed because there's nothing in state to compare to. To rotate a secret, change the value in config and apply; the provider always sends whatever the config currently has.
- State files written by 3.18.0 are forward-compatible with 3.17.x readers (the field is just absent from state), but a 3.17.x reader will complain that the resource is "missing" the previously-required value if you try to plan with the old provider against new state. Pin one direction.

If you previously stored secrets only in state (e.g. relied on `terraform import` to pull `password` back from a long-lived state file), move them to a managed source — SSM/Vault/Terraform Cloud variables/`-var-file` — before upgrading.

## Features

### Provider-level `custom_header` block

The provider now accepts an optional list of `custom_header { name, value }` blocks. Each pair is sent as an HTTP header on the login call and on every authenticated Connect RPC the provider issues. This is intended for fronting environments that put Bytebase behind a zero-trust / access gateway, where requests must carry a token alongside the Bytebase service-account credentials.

```hcl
provider "bytebase" {
  url             = "https://bytebase.example.com"
  service_account = "service@example.com"
  service_key     = var.bytebase_service_key

  custom_header {
    name  = "zero_trust_token"
    value = var.zero_trust_token
  }

  custom_header {
    name  = "x-access-id"
    value = var.gateway_access_id
  }
}
```

Behavior:

- `name` is validated against the HTTP token grammar (`[!#$%&'*+.^_\`|~0-9A-Za-z-]+`) and canonicalized via `http.CanonicalHeaderKey` before sending. `zero_trust_token` and `Zero-Trust-Token` collide on the wire — declare each header once.
- `value` is `Sensitive`, so it does not appear in plan output. Use a variable or `sensitive` source for the value.
- The provider's own `Authorization: Bearer ...` header is authoritative — a `custom_header { name = "Authorization" ... }` is silently overridden by the access token the provider obtains from `service_account` / `service_key`. This is intentional and prevents misconfiguration from breaking auth.
- The block is optional. Existing configurations that don't declare `custom_header` send no extra headers and behave identically to 3.17.4.
