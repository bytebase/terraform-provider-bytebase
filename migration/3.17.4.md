# Migrate from 3.17.0 to 3.17.4

> Aggregates the user-visible changes shipped in 3.17.1, 3.17.2, 3.17.3, and 3.17.4.

## Server requirement

The provider requires a Bytebase server **≥ 3.17.0**. Earlier servers return a bare-UUID `workspace` value (legacy `workspace_id`); against those, `NewClient` now fails fast at provider init with:

```
legacy workspace value "<uuid>" (expected "workspaces/<id>"); this provider version requires Bytebase >= 3.17.0. Upgrade the server, or pin the provider to "~> 3.16".
```

Path-based TLS (see below) additionally requires a Bytebase server **≥ 3.17.1**.

## Bug fixes that may surface as plan diffs

These are corrections, not breaking changes — but each can produce a one-time plan diff after upgrading.

### `bytebase_iam_policy` import now binds to the correct `parent`

Before 3.17.1, importing a project-level IAM policy with `import { id = "projects/foo" }` silently read the **workspace** policy because the importer never set `parent`. After 3.17.1, the importer sets `parent` from the import ID and the read returns the correct project policy. If you previously imported and the imported state happened to match the workspace policy by accident, the next plan after upgrade will show a one-time refresh diff.

### `bytebase_group.members` is now `Optional`

Before 3.17.2 the block was `Required` with an implicit "≥ 1 owner" guard. As of 3.17.3 the block is `Optional`, empty member sets are valid, and the owner guard is removed. SCIM-synced and bootstrapping groups can now be declared without members:

```terraform
resource "bytebase_group" "scim_synced" {
  name        = "groups/synced"
  title       = "SCIM-synced group"
  description = "Members managed externally via SCIM"
  # members intentionally omitted
}
```

Existing groups with members continue to plan/apply unchanged.

## Features

### `object_schema_json` on database catalog tables (3.17.2)

Nested-document masking for OpenSearch, Elasticsearch, and other document-oriented engines is now expressible via Terraform. A new `object_schema_json` attribute on `bytebase_database.catalog.schemas.tables` accepts a JSON string that is parsed through the `v1pb.ObjectSchema` proto for type-aware validation and canonicalized (sorted keys, normalized whitespace) so plans don't flap on formatting differences.

```terraform
resource "bytebase_database" "opensearch_users" {
  name = "instances/opensearch-cluster/databases/node-1"
  # ...

  catalog {
    schemas {
      name = ""
      tables {
        name = "users_index"
        object_schema_json = jsonencode({
          type = "OBJECT"
          structKind = {
            properties = {
              email = { type = "STRING", semanticType = "<uuid-from-ui>" }
            }
          }
        })
      }
    }
  }
}
```

### TLS material via filesystem path on `bytebase_instance` (3.17.4)

Each TLS slot (CA / client cert / client key) on `bytebase_instance.data_sources` accepts **either** an inline PEM string **or** an absolute filesystem path readable by the Bytebase server process. The two variants are mutually exclusive per slot — the server rejects a request that sets both — but mix-and-match across slots is allowed (e.g. inline CA with a path-based client cert/key).

| Inline (existing) | Path (new in 3.17.4) | Configured flag (new, computed) |
| --- | --- | --- |
| `ssl_ca` | `ssl_ca_path` | `ssl_ca_set`, `ssl_ca_path_set` |
| `ssl_cert` | `ssl_cert_path` | `ssl_cert_set`, `ssl_cert_path_set` |
| `ssl_key` | `ssl_key_path` | `ssl_key_set`, `ssl_key_path_set` |

The `*_set` and `*_path_set` flags are returned by the API and surface whether each slot is configured on the server. Inline PEM and path values themselves are write-only (`INPUT_ONLY` on the proto) and are not returned by reads — they're propagated from prior state on the resource and are always empty on the data source.

Inline-PEM example (unchanged behavior):

```terraform
resource "bytebase_instance" "mysql_ssl" {
  resource_id = "mysql-ssl"
  environment = "environments/test"
  engine      = "MYSQL"
  activation  = true

  data_sources {
    id                     = "admin"
    type                   = "ADMIN"
    host                   = "mysql.example.com"
    port                   = "3306"
    username               = "admin"
    password               = "your-password"
    use_ssl                = true
    verify_tls_certificate = true
    ssl_ca                 = file("${path.module}/certs/ca.pem")
    ssl_cert               = file("${path.module}/certs/client-cert.pem")
    ssl_key                = file("${path.module}/certs/client-key.pem")
  }
}
```

Path-based example:

```terraform
resource "bytebase_instance" "mysql_ssl_path" {
  resource_id = "mysql-ssl-path"
  environment = "environments/test"
  engine      = "MYSQL"
  activation  = true

  data_sources {
    id                     = "admin"
    type                   = "ADMIN"
    host                   = "mysql.example.com"
    port                   = "3306"
    username               = "admin"
    password               = "your-password"
    use_ssl                = true
    verify_tls_certificate = true
    ssl_ca_path            = "/etc/bytebase/certs/ca.pem"
    ssl_cert_path          = "/etc/bytebase/certs/client-cert.pem"
    ssl_key_path           = "/etc/bytebase/certs/client-key.pem"
  }
}

# Inspect which TLS slots the server reports as configured.
#
# data_sources is a Set whose elements include sensitive fields (password,
# ssl_ca, etc.), so iterating over it taints the resulting expression.
# nonsensitive() unwraps the bool flags, which are not sensitive themselves.
output "mysql_ssl_path_tls_status" {
  value = nonsensitive({
    for ds in tolist(bytebase_instance.mysql_ssl_path.data_sources) :
    ds.id => {
      ssl_ca_set        = ds.ssl_ca_set
      ssl_cert_set      = ds.ssl_cert_set
      ssl_key_set       = ds.ssl_key_set
      ssl_ca_path_set   = ds.ssl_ca_path_set
      ssl_cert_path_set = ds.ssl_cert_path_set
      ssl_key_path_set  = ds.ssl_key_path_set
    }
  })
}
```

#### Server-side validation tightened in Bytebase 3.17.1

These rules are enforced by the Bytebase server, not by the provider — the provider surfaces the server error verbatim:

- **Mutual exclusion per slot.** Submitting both `ssl_ca` and `ssl_ca_path` (or the corresponding `ssl_cert*` / `ssl_key*` pair) is rejected. To switch from inline PEM to path on an existing instance, clear the inline field in the same apply that sets the path field.
- **PEM validity at save time.** Inline CA/cert/key PEM content is parsed at save time. Malformed PEM that previously only failed at connection time will now fail the apply.
- **Absolute paths only.** `ssl_*_path` values must be absolute paths.
